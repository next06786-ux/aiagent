第一章 软件概述

1.1 软件简介

ParaLife · 平行境（以下简称"本软件"）是一款基于AI智能体架构的鸿蒙原生个性化生活决策助手。软件名称"平行境"寓意：对于用户面临的每一个重要决策，系统都会为其推演出一条平行的人生轨迹，帮助用户在做出选择之前，看清每条路径的可能走向。

本软件运行于华为鸿蒙（HarmonyOS NEXT）操作系统，后端采用Python/FastAPI构建AI服务，前端采用ArkTS/ArkUI实现原生界面。系统集成了大语言模型个性化微调、知识图谱、检索增强生成、数字孪生等前沿AI技术，为用户提供真正"千人千面"的智能决策辅助体验。

1.2 开发背景与目的

当前市场上的AI助手产品普遍存在"千人一面"的问题——无论用户是谁，系统给出的回答和建议都大同小异，无法深度理解个体的价值观、生活习惯、人际关系和情绪状态。这一局限在用户面临人生重大抉择时尤为突出。

本软件的开发目的在于：通过持续学习用户的个性特征，构建专属的个人知识图谱，并以此为基础训练专属AI模型，使系统能够真正理解"这个人"，而非泛化的"所有人"。当用户面临职业转换、升学选择、城市迁移等重大决策时，系统基于对用户的深度理解，对每个决策选项进行12个月的时间线推演，量化评估各路径的风险与收益。

1.3 面向用户

本软件主要面向18-35岁、正处于人生关键节点的青年群体，具体包括：

  • 面临职业选择或转型的应届毕业生与职场人士
  • 考虑升学深造（考研、出国）的在校学生
  • 面临城市迁移、置业等重大生活决策的青年
  • 希望通过AI辅助进行自我认知与成长规划的用户

第三章 功能说明

3.1 个性化知识图谱构建

本软件在用户日常对话过程中，通过 LLM 智能提取（优先）与正则规则匹配（降级）相结合的方式，从对话文本中识别用户提及的人物实体，自动构建并更新个人人际关系知识图谱。

人物关系分为五类：family（家人）、close_friends（好友/闺蜜）、colleagues（同事/领导）、friends（朋友/同学）、weak_ties（弱关系）。每个人物节点包含姓名、关系类别、置信度及描述属性。

知识图谱以 Neo4j 图数据库为存储后端，节点类型为人物（Person），边类型表示人际关系。前端以 3D 星图形式可视化展示，用户可交互探索自己的人际关系网络。

主要操作流程：

  1. 用户发送对话消息
  2. 信息提取器（InformationExtractor）优先调用 LLM（qwen-turbo）从文本中识别人物，失败时降级为正则匹配
  3. 知识图谱系统（InformationKnowledgeGraph）将提取结果写入 Neo4j
  4. 前端 KnowledgeStarMap 页面实时渲染更新后的人际关系图谱

3.2 专属AI模型训练（LoRA）

本软件为每位用户维护一个独立的LoRA适配层，叠加在Qwen3.5-9B基座模型之上，形成专属的个性化AI模型。

训练数据来源：用户对话历史、决策记录、平行人生游戏选择行为、用户反馈等。LoRA调度器（LoRAScheduler）根据配置文件定期触发增量训练，每次训练完成后自动更新用户的LoRA权重文件。

用户可在前端LoRATraining页面查看训练状态、触发手动训练，并通过对比测试感受个性化模型与通用模型的差异。

技术参数：LoRA rank=16，alpha=32，目标模块为q_proj/v_proj/k_proj/o_proj，训练轮次根据数据量自适应调整。

3.3 决策路径推演

当用户面临重要决策时，可在"决策副本"功能中输入决策标题与各备选方案。系统基于用户的个人知识图谱与历史数据，对每个选项进行12个月的时间线推演。

推演维度包括：职业发展轨迹、经济状况变化、社交关系影响、心理健康风险、学习成长机会等。系统为每条路径生成量化的风险评分与收益预测，并以可视化图表形式呈现对比结果。

决策推演流程：

  1. 用户在QuickDecisionInput页面输入决策信息
  2. 后端DecisionEngine调用用户专属LoRA模型进行路径模拟
  3. RiskAssessmentEngine对各路径进行风险量化评估
  4. EnhancedDecisionResult页面展示多维度对比分析报告
  5. 用户可保存决策记录，系统持续跟踪实际结果与预测的偏差

3.4 平行人生决策游戏

平行人生游戏是本软件的特色功能，以沉浸式叙事游戏的形式帮助用户探索不同人生选择的后果。游戏采用事件链结构，玩家扮演主角，在职场、情感、财务等场景中连续做出4-5个决策，每次选择实时影响情绪值、财务值、社交值三个属性，最终走向不同结局。

游戏设计特点：每个选项均标注属性变化预览（如"情绪+10，财务-5"），帮助用户理性权衡；15秒倒计时机制模拟真实决策压力；游戏结束后生成决策画像，分析用户的风险偏好与价值倾向。

游戏数据价值：用户的每次游戏选择都会转化为高质量的对话训练样本，自动存入数据库，用于后续LoRA模型的增量训练，形成"游戏-学习-个性化"的正向循环。

3.5 涌现行为分析与预警

本软件引入复杂系统涌现理论，对用户多维行为数据进行时序分析。涌现检测器（EmergenceDetector）持续监测用户的对话频率、情绪波动、睡眠质量、社交活跃度等指标，识别局部行为模式向全局特征的涌现过程。

当系统检测到潜在的负面涌现（如持续情绪低落、社交退缩的协同出现）时，会主动向用户推送洞察提示，建议关注相关领域。前端EmergenceDashboard页面以可视化方式展示涌现模式与历史趋势。

3.6 多模态数据采集

本软件支持多种数据输入方式，构建全面的用户数字画像：

数据类型采集方式用途
文本对话用户主动输入知识图谱构建、LoRA训练
图片相机拍摄/相册上传场景识别、生活记录
语音麦克风录音语音转文字后处理
传感器数据设备传感器（步数、心率）健康分析
健康数据用户手动填写六大领域综合分析
游戏行为平行人生游戏决策偏好分析、LoRA训练

所有数据经多模态融合系统（MultimodalFusion）处理后，分别存入RAG记忆系统、知识图谱和关系数据库，形成完整的用户数字档案。

3.7 数字孪生预测

本软件为每位用户构建数字孪生模型（DigitalTwin），基于历史数据建立状态转移方程，对用户未来7-30天的生活状态进行轨迹预测。预测维度包括睡眠质量、运动水平、压力指数、社交活跃度、财务稳定性、学习进度等。

用户还可通过"干预模拟"功能，输入计划中的生活改变（如"每天增加30分钟运动"），系统模拟该干预方案的预期效果，与基线轨迹进行对比，帮助用户评估行动计划的可行性。

第四章 技术实现

4.1 LoRA个性化微调技术

LoRA（Low-Rank Adaptation）是一种参数高效的大模型微调方法。其核心思想是：在预训练模型的权重矩阵旁边添加一个低秩分解矩阵，训练时只更新低秩矩阵的参数，而冻结原始权重。

本软件的LoRA实现方案如下：

  • 基座模型：Qwen3.5-9B（通义千问3.5，90亿参数）
  • 微调框架：HuggingFace PEFT 0.18.1
  • 目标模块：注意力层的q_proj、k_proj、v_proj、o_proj
  • LoRA超参数：rank=16，alpha=32，dropout=0.05
  • 训练数据：用户对话历史（自动收集，最少50条触发训练）
  • 训练方式：增量训练，每次训练在上一版本基础上继续优化
  • 模型存储：每用户独立目录，支持量化版本（INT4）节省存储

LoRA调度器（LoRAScheduler）读取配置文件（backend/data/scheduler_config.json），支持定时触发和手动触发两种模式。训练完成后，新模型自动替换旧版本，下次对话即生效。

4.2 融合模型压缩技术（FlatQuant + OBR）

为支持模型在资源受限的移动端高效运行，本软件实现了FlatQuant与OBR（Outlier-Based Rounding）的融合压缩方案。

FlatQuant技术：通过可学习的正交变换矩阵，将权重分布"拉平"，使其更适合低比特量化。在校准训练阶段，系统对每一层的变换矩阵进行优化，最小化量化误差。

OBR技术：针对权重中的离群值（Outlier）进行特殊处理，采用两组残差量化策略，在保持量化精度的同时降低比特宽度。

融合压缩流程（FusionCompressor）：

  1. 加载基座模型并应用FlatQuant包装
  2. 在FlatQuant包装状态下进行校准训练（15 epochs）
  3. 收集变换后的激活统计（Hessian矩阵、激活熵、稳定秩）
  4. 执行FlatQuant重参数化，将变换矩阵融合进权重
  5. 基于Hessian统计执行llmquant SVD低秩分解与残差量化
  6. 保存压缩后的模型（INT4精度，体积约为原始的25%）

压缩效果：模型体积从约18GB压缩至约4.5GB，推理速度提升约3倍，生成质量损失控制在可接受范围内（困惑度增加小于5%）。

4.3 Neo4j知识图谱技术

本软件采用Neo4j图数据库存储用户的个人知识图谱，利用图数据库天然的关系表达能力，高效存储和查询复杂的实体关系网络。

图谱节点类型：

节点类型属性示例
Person（人物）name, category, confidence, description爸爸（family）、张伟（friends）、李经理（colleagues）

人际关系分类：family（家人）、close_friends（好友）、colleagues（同事/领导）、friends（朋友/同学）、weak_ties（弱关系）。

信息提取器（InformationExtractor）优先调用 LLM（qwen-turbo）进行智能提取，失败时降级为正则规则匹配（称谓模式 + 姓名模式 + "叫XX"模式）。提取结果通过 Cypher 查询语言写入 Neo4j，前端通过 3D 星图可视化，节点大小反映关系重要性。

4.4 涌现理论分析技术

涌现（Emergence）是复杂系统理论中的核心概念，指系统整体表现出各部分单独不具备的新特性。本软件将涌现理论应用于用户行为分析：将用户的多维生活数据视为一个复杂系统，通过检测各维度指标的协同变化，识别潜在的行为模式涌现。

涌现检测算法（OptimizedEmergenceDetector）采用滑动窗口时序分析，计算各维度指标的互相关系数与趋势一致性，当多个负面指标同步降低时触发涌现预警。实时分析器（RealtimeAnalyzer）在每次对话后自动运行，提取情绪、话题、意图等特征，持续更新用户的行为模式画像。

5.2 后端部署步骤

第一步：克隆代码仓库并进入项目目录。

第二步：创建Python虚拟环境并安装依赖。执行命令：pip install -r requirements.txt

第三步：配置环境变量。复制 backend/.env.example 为 backend/.env，填写以下关键配置项：

  • MODEL_PATH：本地Qwen3.5-9B模型路径
  • LORA_MODELS_DIR：LoRA模型存储目录
  • NEO4J_URI：Neo4j数据库连接地址
  • NEO4J_PASSWORD：Neo4j数据库密码
  • DASHSCOPE_API_KEY：阿里云灵积API密钥（用于视觉/语音模型）
  • DATABASE_URL：关系数据库连接字符串

第四步：初始化数据库。执行命令：python backend/database/init_db.py

第五步：启动后端服务。执行命令：uvicorn backend.main:app --host 0.0.0.0 --port 8000

第六步：验证服务状态。访问 http://服务器IP:8000/health，返回 status: healthy 表示启动成功。

5.3 前端部署步骤

第一步：安装DevEco Studio 5.0及以上版本，配置HarmonyOS SDK API 12。

第二步：打开 harmonyos/ 目录作为项目根目录。

第三步：修改 harmonyos/entry/src/main/ets/config/ApiConfig.ets 中的服务器地址，填写后端服务的实际IP和端口（开发时使用 VS Code 1.111.0 通过 SSH 远程连接服务器进行调试）。

第四步：连接华为鸿蒙设备（开启开发者模式），点击DevEco Studio的"Run"按钮编译并安装到设备。

第五步：首次启动App，在注册页面创建账号，即可开始使用。

注意：首次启动后端服务时，系统会自动加载Qwen3.5-9B基座模型，该过程需要约2-5分钟，请耐心等待。模型加载完成后，健康检查接口的llm字段将显示"ready"状态。

第七章 数据库设计

7.1 关系数据库（MySQL 8.0）

关系数据库采用 MySQL 8.0，通过 PyMySQL 驱动连接，使用 SQLAlchemy ORM 进行对象关系映射，存储用户账户信息、健康记录、对话历史等结构化数据，主要数据表如下：

表名主要字段说明
usersuser_id, username, email, password_hash, nickname, created_at用户账户信息
health_recordsid, user_id, sleep_hours, sleep_quality, steps, heart_rate, exercise_minutes, stress_level, health_score, timestamp每日健康数据记录
conversation_historyid, user_id, session_id, role, content, thinking, timestamp对话历史（含思维链）
decision_recordsid, user_id, title, options, analysis_result, chosen_option, timestamp决策记录与推演结果
lora_training_logsid, user_id, task_id, status, data_count, started_at, completed_atLoRA训练任务日志
user_feedbackid, user_id, message_id, feedback_type, content, timestamp用户反馈数据

7.2 图数据库（Neo4j）

Neo4j图数据库存储用户个人知识图谱，采用属性图模型。每个用户的图谱数据通过user_id进行隔离，确保数据安全。

主要节点标签：Person、Event、Concept、Place、Skill、Emotion

主要关系类型：KNOWS（认识）、PARTICIPATED_IN（参与）、RELATED_TO（相关）、LOCATED_AT（位于）、HAS_SKILL（具备技能）、FEELS（情绪关联）

查询示例（Cypher）：获取用户的所有重要人际关系节点，按重要性排序，返回前10个。

7.3 向量数据库（FAISS）

FAISS向量数据库用于RAG记忆系统的语义检索。用户的每条记忆（对话、图片描述、传感器摘要等）经文本嵌入模型转换为向量后存入FAISS索引。

记忆类型分类：CONVERSATION（对话记忆）、PHOTO（图片记忆）、SENSOR_DATA（传感器数据）、DECISION（决策记忆）、INSIGHT（洞察记忆）

检索时，系统将用户当前输入转换为查询向量，在FAISS中进行近似最近邻搜索，返回最相关的历史记忆，注入LLM上下文，提升回答的个性化程度。

第八章 安全说明

8.1 用户数据安全

  • 用户密码采用bcrypt算法加密存储，不保存明文密码
  • 用户登录采用Token机制（JWT），Token有效期可配置
  • 不同用户的知识图谱数据通过user_id严格隔离，不同用户间无法互相访问
  • 不同用户的LoRA模型文件存储在独立目录，权限隔离
  • API接口支持Token验证，敏感接口需携带有效Token才能访问

8.2 数据传输安全

  • 生产环境建议启用HTTPS/WSS加密传输
  • API接口支持CORS配置，可限制允许访问的域名
  • 图片、语音等多媒体数据在服务端处理后不长期保存原始文件

8.3 隐私保护

  • 用户个人知识图谱数据仅用于为该用户提供个性化服务，不用于其他目的
  • LoRA训练数据来源于用户自身的对话历史，不与其他用户共享
  • 用户可随时通过账户管理功能删除个人数据
  • 系统不收集设备标识符等敏感隐私信息

8.4 系统稳定性

  • 所有AI模块采用延迟初始化策略，避免启动时阻塞
  • 各模块均有异常捕获机制，单个模块故障不影响整体服务
  • LoRA训练任务在后台异步执行，不影响前台对话服务
  • 数据库操作均有事务保护，确保数据一致性
