# 统一智能体架构设计

## 1. 总体目标

在纯 API 推理模式下，这个系统的核心不应该再定义为“聊天 + 若干功能页”，而应该定义为：

`AI 核心驱动的个性化未来模拟系统`

系统由四个核心部分组成：

1. `AI 核心`
   作为唯一主入口，通过对话统一编排所有能力。
2. `知识星图`
   作为长期记忆与关系底座，提供一张底层图、两种前台视图。
3. `决策图谱舞台`
   作为多 Agent 分支模拟结果的主展示层。
4. `平行人生游戏`
   作为用户真实决策信息的交互采集层。

这四部分不是并列功能，而是一个闭环系统：

`AI Core -> 调取知识星图 -> 生成多分支 Agent -> 输出决策图谱 -> 送入平行人生采集 -> 回写知识星图与个性化画像 -> 继续对话迭代`

## 2. 产品心智

系统最终给用户的感觉不应该是：

`我在几个页面之间切来切去`

而应该是：

`我在和一个理解我、记得我、能帮我模拟未来的 AI 核心对话`

也就是说：

1. 用户永远从 AI 对话进入系统。
2. 知识星图、决策图谱、平行人生都是 AI 核心调出的能力层。
3. 用户和 AI 的每次交互，都会更新系统对用户的长期理解。

一句话定义：

`用户不是在和聊天机器人说话，而是在和自己的未来操作系统说话。`

## 3. 四层统一架构

### 3.1 第一层：AI 核心

AI 核心是系统主控，不只是大模型回答器，而是 `对话式编排器`。

它负责：

1. 理解用户问题
2. 识别当前任务阶段
3. 决定调哪部分知识
4. 决定启动哪些分支 Agent
5. 决定何时切入平行人生
6. 决定何时回收信息并更新画像

### 3.2 第二层：知识星图

知识星图是系统长期记忆底座。

底层是一张混合知识图谱，但前台只开放两种视图：

1. `人物关系视图`
2. `其它类型节点视图`

### 3.3 第三层：决策图谱舞台

决策图谱舞台是 `多 Agent 分支模拟` 的主展示层。

它负责把不同分支 Agent 走出来的未来路径变成图谱舞台，让用户看到：

1. 有哪些路
2. 每条路如何发展
3. 哪些节点是关键转折点
4. 哪些关系和状态正在影响分支

### 3.4 第四层：平行人生游戏

平行人生不是附属娱乐层，而是 `真实决策采集器`。

它通过可体验、可代入、可重复的游戏机制，收集比普通问答更真实的用户决策信号。

这些信号再反哺：

1. 知识星图
2. 个性化画像
3. 决策推演算法
4. AI 核心后续对话策略

## 4. AI 核心的完整职责

AI 核心建议拆成五个内部模块。

### 4.1 Intent Router

负责识别当前用户输入属于哪种任务：

1. 关系探索
2. 决策求助
3. 路径比较
4. 分支体验
5. 结果复盘

### 4.2 Context Builder

负责从知识星图、历史对话、平行人生记录中组装当前任务上下文。

输出：

1. 当前问题摘要
2. 当前状态摘要
3. 关键人物
4. 关键约束
5. 长期目标
6. 高风险点
7. 个性化决策向量

### 4.3 Dialogue Manager

负责控制对话流程。

建议把系统对话状态分成五类：

1. `Discover`
   发现问题、补充信息。
2. `Model`
   建立当前状态模型。
3. `Simulate`
   展开多分支推演。
4. `Experience`
   将某条分支送入平行人生。
5. `Reflect`
   复盘结果并回写画像。

### 4.4 Tool Orchestrator

负责调度各能力层：

1. 调知识星图取子图
2. 调多 Branch Agent 生成分支
3. 调决策图谱舞台输出结果
4. 调平行人生游戏生成可交互剧情

### 4.5 Personalization State Manager

负责维护用户个性化决策画像。

建议维护一组可持续更新的个性化维度：

1. `risk_tolerance`
   风险承受度
2. `delay_discount`
   短期收益偏好
3. `social_dependency`
   关系依赖度
4. `execution_stability`
   执行稳定性
5. `growth_bias`
   长期成长偏好
6. `loss_aversion`
   损失厌恶程度
7. `ambiguity_tolerance`
   不确定性容忍度

这些维度不只来自问答，而是来自：

1. 历史对话
2. 决策记录
3. 平行人生行为
4. 后续回访结果

## 5. 知识星图设计

## 5.1 一张底层图，两种视图

底层是一张混合图谱，包含：

1. 人物节点
2. 关系节点
3. 状态节点
4. 事件节点
5. 决策节点
6. 目标节点
7. 风险节点
8. 资源节点
9. 未来状态节点

但前台只开放两种投影视图，避免图谱过乱。

### 视图一：人物关系视图

默认视图。

重点展示：

1. 用户与关键人物关系
2. 关键人物之间的关系
3. 哪些人物正在影响当前决策
4. 支持者、阻碍者、压力源、资源源分别是谁

这是最适合作为知识星图首页的视图，因为它最容易理解，也最有情感代入感。

### 视图二：其它类型节点视图

用于进入更深的决策建模。

重点展示：

1. 当前状态
2. 决策上下文
3. 关键事件
4. 关键约束
5. 风险与资源
6. 未来状态锚点

这个视图不是全量展开，而是围绕当前问题只展示高相关子图。

## 5.2 知识星图的算法职责

知识星图不只是存储，而是要对决策推演和平行人生提供实时支撑。

建议它承担四类算法。

### A. 图谱抽取与写回算法

从以下来源抽取结构化节点与关系：

1. AI 对话
2. 决策推演结果
3. 平行人生行为
4. 用户回访

写回原则：

1. 事实写事实
2. 偏好写偏好
3. 状态写状态
4. 推演结果写成未来状态锚点
5. 不把未验证推测当成已知事实

### B. 子图召回算法

面对一个问题，知识星图不能全图直出，而要召回“当前最相关子图”。

建议相关度得分：

```text
subgraph_score =
0.35 * semantic_relevance +
0.20 * recency +
0.20 * decision_impact +
0.15 * graph_centrality +
0.10 * emotional_salience
```

解释：

1. `semantic_relevance`
   和当前问题语义相关
2. `recency`
   最近是否被提及或更新
3. `decision_impact`
   是否对当前决策结果影响大
4. `graph_centrality`
   在用户图谱中是否重要
5. `emotional_salience`
   是否带强情绪权重

### C. 人物影响传播算法

人物关系不只是展示用，它要影响分支模拟和游戏剧情。

建议每个关键人物计算：

```text
person_influence_score =
relation_strength *
decision_relevance *
emotional_weight *
resource_control_weight
```

这样可以算出：

1. 谁是支持者
2. 谁是阻力源
3. 谁会改变用户执行概率
4. 谁应该出现在平行人生剧情中

### D. 非人物节点影响算法

其它类型节点同样需要参与决策推演。

例如：

1. 状态节点影响起始状态
2. 风险节点影响分支坍塌概率
3. 资源节点影响可执行性
4. 目标节点影响分支排序
5. 事件节点影响转折生成

建议输出一个统一的 `Decision Context Vector`，供分支模拟调用。

## 6. 决策图谱舞台设计

决策图谱舞台的核心不是“评审型多智能体”，而是：

`多智能体分支模拟`

也就是说：

1. 一个 Agent 代表一种走法
2. 一个 Agent 负责往前走出一条人生线
3. 不同 Agent 产生不同未来线程

## 6.1 多 Agent 分支结构

建议决策推演层拆成六个算法模块。

### A. Branch Planner

负责从当前问题中决定需要展开哪些主分支。

分支来源可以有三种：

1. 用户明确提出的选项
2. AI 根据知识星图推断出的隐含选项
3. 从用户历史模式中生成的典型走法

建议主分支数量控制在 3 到 5 条。

### B. Branch Agent Cluster

每个 Agent 只负责一条路径。

例如：

1. 保守稳定线 Agent
2. 激进突破线 Agent
3. 关系优先线 Agent
4. 成长优先线 Agent
5. 观察缓冲线 Agent

每个 Agent 的职责：

1. 基于当前状态向前推进
2. 生成 3 到 6 个关键未来节点
3. 输出每一步的状态变化
4. 输出该分支的关键人物和关键转折点

每个 Agent 不负责：

1. 最终裁决
2. 全局总结
3. 无边界发散

### C. World State Engine

这是分支模拟质量的关键。

如果只有 Agent 写故事，很容易变成幻想小说。

所以所有 Branch Agent 都必须共享同一套 `状态演化引擎`。

建议它维护一个统一状态向量：

1. energy
2. confidence
3. finance
4. social_stability
5. health
6. growth
7. stress
8. optional_domain_axes

Branch Agent 只能在这套共享规则里推进状态。

### D. Transition Model

每一步转移都要显式输出：

1. `state_before`
2. `event`
3. `state_after`
4. `impact_vector`
5. `probability`
6. `execution_confidence`
7. `collapse_risk`

建议节点转移算法：

```text
state_after =
state_before +
event_delta +
relationship_delta +
resource_delta +
habit_delta
```

其中：

1. `event_delta`
   当前事件直接影响
2. `relationship_delta`
   关键人物带来的作用
3. `resource_delta`
   资源约束或资源支持
4. `habit_delta`
   用户长期行为模式带来的惯性

### E. Branch Merge and Prune

不是所有分支都值得展示。

要有一个分支裁剪层，负责：

1. 删掉重复分支
2. 删掉明显不可信分支
3. 删掉偏离用户问题的分支
4. 保留最具差异化和代表性的分支

建议分支得分：

```text
branch_score =
goal_fit +
constraint_fit +
personalization_fit +
diversity_bonus -
collapse_penalty
```

### F. Graph Builder

把多个分支 Agent 的输出组装成统一图谱。

负责：

1. 统一节点格式
2. 统一父子关系
3. 标记 branch_id
4. 合并重叠事件
5. 输出前端可展示的决策图谱

## 6.2 决策节点 Schema

建议决策图谱统一到下面结构：

```json
{
  "branch_id": "branch_growth_01",
  "branch_strategy": "成长优先线",
  "event_id": "node_3",
  "parent_event_id": "node_2",
  "node_level": 3,
  "month": 4,
  "state_before": {
    "energy": 0.61,
    "confidence": 0.55,
    "finance": 0.42
  },
  "event": "减少短期收益，投入长期能力建设",
  "state_after": {
    "energy": 0.52,
    "confidence": 0.67,
    "finance": 0.31
  },
  "impact_vector": {
    "growth": 0.82,
    "finance": -0.23,
    "emotion": 0.14
  },
  "probability": 0.63,
  "execution_confidence": 0.58,
  "collapse_risk": "medium",
  "key_people": ["mentor_a", "friend_b"],
  "evidence_sources": ["kg:habit_learning", "memory:decision_014"]
}
```

这套结构的关键点是：

1. 它能被前端图谱舞台直接渲染
2. 它能被平行人生继续消费
3. 它能写回知识星图作为未来状态锚点

## 6.3 决策图谱舞台的展示算法

舞台不仅需要数据算法，也需要布局算法。

建议前端布局采用：

`中心核心 + 分支星臂 + 时间曲线 + 局部子分支偏移`

具体规则：

1. 中心放当前自我与当前问题
2. 第一圈放主分支入口
3. 每个主分支沿一条曲线向外推进
4. 子分支在主分支侧边偏移展开
5. 时间越往后，节点越向外

建议视觉编码：

1. 节点大小 = `branch_value`
2. 节点发光 = `probability * evidence_strength`
3. 连线粗细 = `execution_confidence`
4. 连线波动 = `uncertainty`
5. 节点颜色 = 主要影响轴
6. 节点粒子噪声 = `collapse_risk`

这样舞台就不仅“好看”，而且每个视觉元素都有语义。

## 7. 平行人生游戏设计

平行人生游戏的核心不是娱乐，而是：

`把用户真实决策倾向从“嘴上说”变成“行为表现”`

## 7.1 游戏的系统定位

平行人生要承担三个职责：

1. 体验某条分支
2. 收集用户决策信息
3. 回写系统以增强个性化

## 7.2 游戏进入方式

建议平行人生有两种进入方式。

### A. 从决策图谱舞台进入

用户点开某条分支或某个关键节点后，进入对应剧情。

这是主路径。

### B. 从 AI 核心主动触发

当 AI 发现信息不足，无法稳定判断用户真实取舍时，可以主动说：

`我们来走一个两分钟的情境模拟，我想看看你在真实压力下会怎么选。`

这时平行人生就成了 `信息补全工具`。

## 7.3 游戏结构设计

建议用 `短回合、高压缩、强取舍` 的方式设计，而不是长篇 RPG。

推荐一局由 4 到 8 个节点组成。

每个节点结构：

1. 情境描述
2. 关键人物登场
3. 资源与风险提示
4. 2 到 4 个选项
5. 隐性代价
6. 结果反馈
7. 情绪反应采集

### 推荐三种玩法模板

#### 模板一：关键抉择链

适合收集：

1. 风险偏好
2. 长短期取舍
3. 对成长和稳定的偏向

#### 模板二：关系冲突链

适合收集：

1. 家庭影响权重
2. 亲密关系影响权重
3. 自我意愿与关系期待冲突下的选择

#### 模板三：执行耐力链

适合收集：

1. 用户会不会坚持
2. 在挫折出现后是否回退
3. 用户对长期路径的执行稳定性

## 7.4 平行人生采集哪些数据

平行人生不能只记录“选了哪个选项”，而要记录更深层的行为信号。

建议采集：

1. `choice_path`
   用户走过的选项路径
2. `decision_latency`
   决策时长
3. `reversal_count`
   是否频繁反悔
4. `abandon_rate`
   是否容易中途退出
5. `risk_preference_signal`
   面对高收益高风险是否更愿意选
6. `relationship_weight_signal`
   面对人物压力是否改变选择
7. `stress_response_signal`
   高压情境下是否更保守
8. `execution_persistence_signal`
   遇到挫折时是否继续推进
9. `emotion_feedback`
   用户对结果的主观情绪评价

## 7.5 游戏数据如何喂给决策推演

这些行为数据不能只存档，必须转成可用于推演的结构化画像。

建议生成一个 `Decision Behavior Profile`：

1. `actual_risk_tolerance`
2. `actual_social_dependency`
3. `actual_execution_stability`
4. `actual_growth_bias`
5. `actual_loss_aversion`
6. `actual_recovery_after_setback`

然后把它用于：

1. 更新 `execution_confidence`
2. 更新分支排序
3. 更新人物影响权重
4. 更新资源压力阈值
5. 更新平行人生下一轮剧情生成策略

## 8. 三大功能如何真正打通

## 8.1 知识星图如何影响决策推演

### 人物关系的影响

人物关系要直接进入分支模拟。

例如：

1. 导师支持会提高成长线成功率
2. 家庭压力会提高稳定线吸引力
3. 伴侣反对会降低激进线执行概率
4. 朋友资源会提高创业线资源得分

也就是说，人物关系不是展示层素材，而是 `分支状态转移因子`。

### 其它类型节点的影响

其它类型节点也要进入决策推演。

例如：

1. 当前财务状态影响可执行选项
2. 当前精力和健康影响分支持续性
3. 当前长期目标影响分支优先级
4. 风险节点影响坍塌概率
5. 关键历史事件影响当前行为惯性

## 8.2 知识星图如何影响平行人生

知识星图应该决定平行人生里的：

1. 哪些人物出场
2. 哪些冲突最真实
3. 哪些资源约束必须出现
4. 哪些情境最值得模拟

例如：

1. 人物关系权重高的人，更容易被写进剧情冲突
2. 当前最关键的状态约束，更应该体现在游戏选项代价上
3. 历史上用户反复纠结的问题，应成为高优先级剧情模板

## 8.3 平行人生如何影响知识星图

游戏结束后，不只记录“玩过一次”，而应写回：

1. 新的人物影响边权
2. 新的偏好节点
3. 新的执行倾向节点
4. 新的风险阈值
5. 新的未来状态偏好

这样知识星图就会越来越接近“真实用户”，而不是“口头用户”。

## 8.4 AI 核心如何贯穿三者

AI 核心应该始终做这四件事：

1. 调子图
2. 展分支
3. 派剧情
4. 收反馈

也就是说，AI 核心不是旁白，而是整个系统的总导演。

## 9. 个性化算法体系

系统的个性化不能只靠 prompt，要靠持续更新的算法层。

建议分成四套个性化算法。

## 9.1 用户状态建模算法

目标：

构建“当前用户是谁、处于什么阶段”的动态状态向量。

输入：

1. 对话内容
2. 知识星图节点
3. 历史决策
4. 平行人生行为

输出：

1. 当前状态向量
2. 决策域权重
3. 当前脆弱维度
4. 当前优势维度

## 9.2 分支个性化生成算法

目标：

不是生成“所有可能人生”，而是生成“最像这个用户真的会走出的几条人生”。

建议分支种子得分：

```text
seed_score =
option_relevance +
goal_alignment +
constraint_compatibility +
relationship_support +
behavior_profile_fit
```

## 9.3 执行概率算法

目标：

判断某条路用户真的会不会走下去。

初期可以用规则加权，后期可训练小评分器。

建议输入：

1. 平行人生执行稳定性
2. 历史相似任务完成率
3. 当前资源余量
4. 当前关系阻力
5. 当前压力承受能力

输出：

`execution_confidence`

## 9.4 分支可信度与展示优先级算法

目标：

决定哪些分支值得进入主舞台，哪些只作为次级分支折叠显示。

建议：

```text
display_priority =
branch_score *
execution_confidence *
user_attention_weight
```

## 11. 端到端工作流

建议系统完整流程如下：

1. 用户通过 AI 对话提出问题
2. AI 核心识别这是决策任务
3. Context Builder 从知识星图召回人物关系子图和状态子图
4. AI 核心先展示人物关系视图，让用户快速进入上下文
5. 需要更深建模时，切换到其它类型节点视图
6. AI 核心调用 Branch Planner 生成 3 到 5 条主分支
7. 多个 Branch Agent 分别模拟未来路径
8. World State Engine 统一推进状态变化
9. Graph Builder 生成决策图谱舞台
10. 用户在舞台上对比各条路
11. 用户点击某个关键分支进入平行人生
12. 平行人生收集真实决策信号
13. Decision Behavior Profile 更新
14. 知识星图回写新边权和新偏好节点
15. AI 核心进入下一轮更个性化的对话与推演

## 12. MVP 落地顺序

建议按下面顺序推进。

### 第一阶段

1. 知识星图收口为一张底层图、两种前台视图
2. AI 核心接入对子图视图的调度
3. 决策图谱舞台统一成多 Agent 分支展示

### 第二阶段

1. 接入 World State Engine
2. 接入 Branch Merge and Prune
3. 平行人生改造成分支体验与行为采集工具

### 第三阶段

1. 打通平行人生到知识星图的回写
2. 打通行为画像到分支排序与执行概率
3. 实现真正的个性化分支模拟闭环

## 13. 最终结论

这套系统最合理的统一方式是：

1. `知识星图`
   负责长期记忆与关系建模，并提供两种视图
2. `决策图谱舞台`
   负责多 Agent 分支模拟结果展示
3. `平行人生游戏`
   负责采集真实决策行为
4. `AI 核心对话`
   作为唯一主入口，统一编排全部流程

它们不是三个功能加一个聊天框，而是一个闭环智能体架构：

`AI Core -> Graph Memory -> Branch Agent Cluster -> Decision Graph Stage -> Parallel Life -> Personalization Update -> AI Core`

这才是最符合你系统气质的完整个性化智能体架构。
