## 🏗️ 核心融合点

### 1. bitsandbytes 推理集成
**用途：** 基座模型动态量化加载

```python
# llm_service.py 中的自动选择逻辑
if enable_quantization and load_in_4bit:
    bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_use_double_quant=True,  # 双量化
        bnb_4bit_quant_type="nf4",        # NormalFloat4
        bnb_4bit_compute_dtype=torch.bfloat16
    )
```

**优点：**
- ✅ 开箱即用，无需预压缩
- ✅ 推理时自动量化
- ✅ 与 PEFT + LoRA 完全兼容
- ✅ 支持异构加载（部分层低精度）

---

### 2. llmquant LoRA 量化
**用途：** 训练后的 LoRA 权重 4-bit 量化

```python
# auto_lora_trainer.py 中的自动量化
await self._auto_quantize_lora_after_training(final_path)
    # ↓
# lora_quantizer.py 调用 llmquant
from external_repos.llmquant.binary import iterative_affine_quantize_perchannel

quantized, info = iterative_affine_quantize_perchannel(
    lora_weight,
    bits=4,
    n_iter=5,      # 5 次迭代优化
    per_channel=True,
    return_info=True
)
```

**优点：**
- ✅ 迭代优化，精度最高
- ✅ Per-channel 量化，与 LoRA rank 完美配合
- ✅ 保存量化元数据（scale/offset）供推理使用
- ✅ 支持 fallback（量化失败→使用原始版本）

**缺点：**
- ⚠️ 需要 calibration（略慢，但后台执行）
- ⚠️ 库 API 要求自行处理反量化

---

### 3. OBR 离线压缩
**用途：** 可选的离线 W4A4KV4+50% 稀疏化压缩

```python
# compress_base_model.py 提供两个方案
# 方案 1（推荐）：bitsandbytes（快速）
result = compressor.compress_with_bitsandbytes()

# 方案 2（可选）：OBR FlatQuant（精度最高）
result = compressor.compress_with_obr()
```

**OBR 的角色：**
- ✅ 完全离线预压缩（一次性）
- ✅ 联合量化+稀疏化（新技术）
- ✅ 4 倍压缩 + 50% 稀疏化
- ✅ 支持 PPL 评估

**与项目的结合：**
- 作为"高精度选项"存在
- 大多数场景使用 bitsandbytes（足够好）
- 特殊场景（超小机器）用 OBR

## 🔄 数据流设计

### 基座模型加载流
```
model_name (HF)
    ↓
llm_service._initialize()
    ├→ 检查 quantization_config
    ├→ enable_quantization=True? YES
    │   ├→ 创建 BitsAndBytesConfig
    │   └→ AutoModelForCausalLM.from_pretrained(..., quantization_config=bnb_config)
    │
    └→ 完成，模型已自动 4-bit 量化加载
```

**显存：** 20 GB → 3-5 GB

---

### LoRA 训练 & 量化流
```
用户对话 (多轮)
    ↓
存入 ConversationHistory (DB)
    ↓
自动检查触发条件 (数据足够? 时间足够?)
    ↓
启动 AutoLoRATrainer.train_lora()
    ├→ 加载基座模型（已量化）
    ├→ 添加 LoRA 适配器
    ├→ 训练（梯度累积）
    ├→ 保存 LoRA 权重 (safetensors)
    │
    └→ 训练完成
        ↓
        自动调用 _auto_quantize_lora_after_training()
        ├→ LoRAQuantizer.quantize_lora_weights()
        │   ├→ 导入 llmquant
        │   ├→ iterative_affine_quantize_perchannel()
        │   ├→ 迭代 5 次优化
        │   └→ 保存量化权重 + 元数据
        │
        └→ 量化完成 (4x 压缩)
```

**存储：** LoRA 50 MB → 12 MB

---

### 推理加载流
```
推理请求 (user_id, prompt)
    ↓
lora_model_manager.load_user_lora(user_id)
    ├→ 检查量化版本是否存在
    │   └→ YES: 加载 user_id_q4/adapter_model.safetensors
    │   └→ NO: 继续
    │
    ├→ 检查原始版本
    │   └→ 加载 user_id/v{N}/final/adapter_model.safetensors
    │
    └→ 如果两个都不存在：使用基座模型
    
    ↓
使用 PEFT 将 LoRA 附加到基座模型
    ↓
推理
    ↓
返回结果
```

**特点：** 自动选择最优版本，无缝 fallback

## 🎨 设计原则

### 1. 正交分离 (Orthogonal Separation)
- 基座量化（bitsandbytes）与 LoRA 量化（llmquant）完全独立
- 基座模型的推理流与训练流分离
- 量化与业务逻辑无耦合

### 2. 渐进降级 (Graceful Degradation)
```
优先级 1: 量化 LoRA + 量化基座
优先级 2: 原始 LoRA + 量化基座
优先级 3: 原始 LoRA + 原始基座
```

### 3. 元数据驱动 (Metadata-Driven)
- 每个量化版本保存元数据（scale, offset, MSE）
- 推理时可选择使用精确量化或近似
- 支持在线调试和性能分析

### 4. 开放式架构 (Pluggable Backend)
- 通用 `Quantizer` 接口支持多种后端
- 易于替换或扩展量化方案
- 不与特定库强绑定

## 🔐 可靠性保证

### 容错机制
1. **推理时容错**
   - 量化 LoRA 加载失败 → 原始 LoRA
   - 原始 LoRA 加载失败 → 基座模型

2. **训练时容错**
   - 量化失败 → 保留原始版本
   - 异常捕获 + 日志记录

3. **推理时容错**
   - 量化推理出错 → fallback 到原始精度
   - 自动告警

### 监控覆盖
```
PPL 变化 ┐
显存使用 ├→ 自动告警
推理延迟 │
成本估算 ┘
```
