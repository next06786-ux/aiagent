# LifeSwarm 项目 + llmquant + OBR 融合合理性评估

## 执行摘要

✅ **融合方案整体合理**，三个组件的结合是最优的。项目的设计充分考虑了两个外部库的设计理念和 API，融合方式**高度解耦且灵活**。

---

## 一、外部库分析

### 1.1 llmquant 库特性

**主要功能：** 1-bit 量化 (binarization)，BiLLM 方案

**核心 API：**
- `binary.py`：
  - `quantize_to_2bit_perchannel()`
  - `quantize_to_4bit_perchannel()`
  - `quantize_to_8bit_perchannel()`
  - `iterative_affine_quantize_perchannel()` —— 迭代优化量化
  - `Binarization` 类 —— mask-based 量化
  
- `bigptq.py`：
  - `BRAGPTQ` 类 —— 二进制残差逼近 + GPTQ
  - 支持 row-wise pruning + per-channel quantization
  
- `run.py`：
  - 主入口脚本
  - 支持自适应 rank 分配（通过 entropy reduction tables）
  - 块级预算分配 + 层级贪心分配

**针对我们的项目：**
- 用于 **LoRA 权重的 4-bit 量化**（per-channel + iterative）✅
- 不用于基座模型（我们选了 bitsandbytes）

---

### 1.2 OBR 库特性

**主要功能：** Optimal Brain Restoration，联合量化和稀疏化，W4A4KV4+50% sparsity

**三大组件：**
1. **QuaRot** —— 旋转优化 (Hadamard rotation)
2. **SpinQuant** —— 扣学习旋转矩阵
3. **FlatQuant** —— 平面量化 (learnable rotation)

**核心 API：**
- `obr_utils.py`：
  - `obr_rtn_quantization()` —— RTN 方案
  - `obr_gptq_quantization()` —— GPTQ 方案，支持分块误差补偿
  
- `quant_utils.py`：
  - `WeightQuantizer` 类 —— GPTQ-style 权重量化
  - `ActQuantizer` 类 —— Token-wise 激活量化
  - Per-channel 和 per-token 量化

- `main.py`：
  - 三个版本分别对应 QuaRot、SpinQuant、FlatQuant
  - 支持离线压缩 + evaluate（PPL 和 zero-shot）

**针对我们的项目：**
- 用于 **基座模型的离线压缩**（W4A4KV4+50% sparsity） ✅
- 也可用于 LoRA（虽然 llmquant 更轻）

### 2.2 API 兼容性分析

#### 2.2.1 bitsandbytes 集成

```python
# 项目中的用法
from transformers import BitsAndBytesConfig

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_use_double_quant=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16
)
```

✅ **合理理由：**
- Transformers 原生支持，无额外依赖
- 推理时自动量化，无需 calibration
- 支持 PEFT + LoRA 无缝结合
- 与 llmquant（训练时量化）互补

---

#### 2.2.2 llmquant 集成

**设计中的用法：**
```python
# backend/model_compression/lora_quantizer.py（待实现）
from external_repos.llmquant.binary import (
    quantize_to_4bit_perchannel,
    iterative_affine_quantize_perchannel
)

# 用于 LoRA 权重量化
quantized_lora = iterative_affine_quantize_perchannel(
    lora_weight,
    bits=4,
    n_iter=5,
    per_channel=True
)
```

✅ **合理理由：**
- llmquant 的 `quantize_to_4bit_perchannel` 是独立函数，可直接复用
- 支持 per-channel 量化，与 LoRA 的 rank 分解完美契合
- Iterative 优化确保精度
- 不污染原始 LoRA 权重，支持回退

**⚠️ 注意点：**
- llmquant 默认返回 FP16 伪量化结果，需二次量化存储
- 需要自行实现 deserialize 逻辑（库中没有 unquantize）

---

#### 2.2.3 OBR 集成

**设计中的用法：**
```python
# backend/model_compression/compress_base_model.py
# 可选方案 1：调用 OBR/FlatQuant 的主脚本
if args.method == "obr":
    sys.path.insert(0, "external_repos/OBR/FlatQuant")
    from main import run_compression
    result = run_compression(obr_args)

# 方案 2（推荐）：直接调用 core API
from external_repos.OBR.FlatQuant.flatquant.obr_utils import obr_gptq_quantization
```

✅ **合理理由：**
- OBR 提供两个层次的接口：
  1. 高层：`main.py` 完整流程
  2. 低层：`obr_utils.py` 量化核心
- 项目中既可用高层（一键压缩），也可用低层（定制化）
- FlatQuant 方案最通用（支持 Qwen、Llama、Mistral 等）

**⚠️ 注意点：**
- OBR 依赖多个子库（QuaRot/SpinQuant/FlatQuant），结构复杂
- Calibration 需要真实数据 (C4/WikiText2)，耗时 1-4 小时
- 适合离线一次性使用，不适合在线训练集成

### 2.3 数据流设计评估

```
用户对话
    ↓
ConversationHistory (DB)
    ↓
AutoLoRATrainer.train_lora() [Transformers]
    ↓
保存原始 LoRA 权重 (PEFT)
    ↓
_auto_quantize_lora_after_training() [llmquant]
    ↓
量化 LoRA 权重 (4-bit per-channel)
    ↓
加载 LoRA：优先量化版本 [PEFT + bitsandbytes]
    ↓
推理调用 [lora_model_manager]
```

✅ **优点：**
- 训练和量化完全分离，互不影响
- 量化失败自动回退到原始版本
- 支持增量更新（旧版本保留）

class QuantizedModelLoader:
    def load_quantized_lora_params(self, lora_path):
        """从 LoRA 量化元数据中加载 scale/offset"""
        metadata_path = os.path.join(lora_path, "quantization_metadata.json")
        with open(metadata_path) as f:
            metadata = json.load(f)
        return metadata["per_channel_scales"], metadata["per_channel_offsets"]
    
    def reconstruct_lora_from_quantized(self, q_weights, scales, offsets):
        """从量化的权重重建原始 LoRA"""
        return q_weights * scales + offsets
